局域网 Web 端 — 架构方案对比

选择 Java 端与前端之间的通信和渲染架构

A

WebSocket 推送 + Canvas 前端渲染（推荐）

Java 端内嵌 WebSocket 服务器 + 简易 HTTP 文件服务器。引擎分析结果实时通过 WebSocket 推送 JSON 数据到前端，前端用 HTML5 Canvas 渲染棋盘、候选点、变化图、热力图。

优点

  • 实时性好 — 引擎每次输出都能即时推送
  • 前端独立渲染 — 悬停变化图等交互在客户端完成，不增加 Java 端负担
  • 多客户端天然支持 — WebSocket 广播
  • 利用已有 Java-WebSocket 依赖，零额外 Java 依赖
  • 前端静态文件打包在 jar 里，部署简单

缺点

  • 需要写前端 Canvas 渲染代码（约 1000-1500 行 JS）
  • 前后端需要定义 JSON 协议
数据流
KataGo 引擎 → GTP Leelaz.java → JSON WebSocket 服务器 → ws:// 浏览器 Canvas
B

服务端渲染截图 + 轮询

Java 端复用 BoardRenderer 的绘制逻辑，将棋盘渲染为 PNG 图片，前端通过 HTTP 轮询获取最新图片。悬停变化图也由服务端渲染并返回。

优点

  • 前端极简 — 只需展示图片
  • 复用已有 Swing 渲染代码

缺点

  • 延迟高 — 轮询模式，非实时
  • 悬停交互困难 — 每次悬停都要请求服务端渲染，延迟明显
  • 带宽消耗大 — 每秒推送多张 PNG
  • 多客户端时 CPU 压力大（每个客户端独立渲染）
  • 移动端适配差 — 图片尺寸固定
C

混合方案：WebSocket 数据推送 + SVG 渲染

与方案 A 类似用 WebSocket 推送数据，但前端用 SVG 而非 Canvas 渲染棋盘。SVG 的 DOM 元素天然支持事件和动画。

优点

  • SVG 元素可直接绑定事件 — 悬停交互更自然
  • 矢量图形 — 任意缩放不失真，适配各种屏幕
  • 可用 CSS 控制样式和动画

缺点

  • 大量 DOM 元素时性能不如 Canvas
  • 热力图等复杂渲染用 SVG 较笨重
  • 19×19 棋盘 + 候选点 + 变化 = 数百个 DOM 节点，可能卡顿

我的推荐：方案 A

WebSocket + Canvas 是这类实时可视化应用的标配方案。项目已有 Java-WebSocket 依赖可以直接用，Canvas 在高频更新和复杂渲染（热力图、变化图）场景下性能最好，且前端交互（悬停变化图）完全在客户端完成，不给 Java 端增加负担。
