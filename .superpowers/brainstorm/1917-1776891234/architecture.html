局域网 Web 端 — 架构方案对比

选择 Java 端与前端之间的通信和渲染架构

A

WebSocket 推送 + Canvas 前端渲染（推荐）

Java 端内嵌 WebSocket 服务器 + 简易 HTTP 文件服务器。引擎每次输出分析结果时，通过 WebSocket 广播 JSON 到所有连接的浏览器客户端。前端用 HTML5 Canvas 渲染棋盘、候选点、变化图、热力图。

优点

  • 实时性最好 — 引擎每次输出都即时推送
  • 悬停变化图等交互完全在客户端完成，不增加 Java 端负担
  • 多客户端天然支持 — WebSocket 广播
  • 利用已有 Java-WebSocket 依赖，零额外 Java 依赖
  • Canvas 渲染高频更新和复杂图形（热力图）性能最佳
  • 前端静态文件打包进 jar，部署零配置

缺点

  • 需要写前端 Canvas 渲染代码（~1000-1500 行 JS）
  • 前后端需要定义 JSON 消息协议

B

服务端渲染截图 + HTTP 轮询

Java 端复用 BoardRenderer 的 Swing 绘制逻辑，将棋盘渲染为 PNG 图片。前端通过 HTTP 每隔 200ms 轮询最新图片。悬停变化图需要前端发送鼠标坐标到服务端，服务端渲染后返回。

优点

  • 前端极简 — 只展示 <img> 标签
  • 复用现有 Swing 渲染代码，视觉效果完全一致

缺点

  • 延迟高 — 轮询间隔 + 图片编码 + 网络传输
  • 悬停变化图延迟明显（鼠标→服务端渲染→返回 PNG）
  • 带宽消耗大 — 每秒多张 PNG 图片
  • 多客户端时 CPU 压力大 — 每个客户端要独立渲染
  • 移动端适配差 — 图片尺寸固定，无法响应式
  • Swing 渲染依赖 AWT 线程，并发竞争风险
C

WebSocket + SVG 前端渲染

与方案 A 类似通过 WebSocket 推送 JSON 数据，但前端用 SVG 而非 Canvas 渲染。SVG 的 DOM 元素天然支持事件绑定和 CSS 动画。

优点

  • SVG 元素可直接绑定鼠标事件 — 悬停检测更简单
  • 矢量图形 — 任意缩放不失真，天然适配各种屏幕
  • 可用 CSS 过渡和动画

缺点

  • 19×19 棋盘 + 候选点 + 变化图 + 热力图 = 数百个 DOM 节点
  • 高频更新时 DOM 操作比 Canvas 慢
  • 热力图等像素级渲染用 SVG 笨重
  • 复杂场景下移动端可能卡顿

💡 推荐：方案 A — WebSocket + Canvas

这是实时可视化应用的标配方案。项目已有 Java-WebSocket 依赖可以直接用；Canvas 在高频更新和复杂渲染（热力图、大量候选点）场景下性能最好；所有交互（悬停显示变化图）在客户端本地完成，完全不增加 Java 端负担；前端打包进 jar 后用户无需任何额外配置。
